# Phase 17 — Requirements

## Purpose

Multiple browser tabs viewing the same session should see the same
conversation, approval requests, pending state, and session
metadata without manual refresh. The two-streams design (per-message
SSE for live tokens + central SSE for state changes, owned by a
SharedWorker) is the chosen architecture.

## Users / actors

- **Tab A** — first tab; opens a session, sends a user message.
- **Tab B** — second tab on the same session; passive viewer.
- **The SharedWorker** — one per origin; the single subscriber to
  the central SSE; broadcasts events to its connected tabs.
- **The server** — single source of truth (sessions on disk, plus
  the in-flight `SessionRuntime` per turn).

## Functional requirements

### Central SSE server endpoint (`GET /api/sync`)

- FR-1. The server exposes a new endpoint `GET /api/sync` that
  opens a persistent SSE response. The connection lives until
  the client disconnects.
- FR-2. Events emitted on the central SSE are restricted to:
  `message_appended`, `session_renamed`, `session_meta_updated`,
  `approval_required`, `tool_result`, `message_done`, `error`.
- FR-3. The central SSE carries **no per-turn streaming events**
  (`message_start`, `token`, `tool_call`, `done`). Those stay on
  the per-message SSE only.

### `message_appended` event

- FR-4. After every successful `appendMessage` (both the user
  message at POST arrival AND the assistant message after a turn),
  the server broadcasts:
  ```ts
  {
    type: "message_appended";
    sessionId: string;
    message: UiMessage;
    originator: string;       // the X-CW-Tab UUID of the originating tab
    ts: string;               // ISO-8601
  }
  ```
- FR-5. The originator is read from the `X-CW-Tab` header on the
  POST that wrote the message; falls back to `"anonymous"` if
  missing. The header is set by the originating tab and read by
  the server.

### `InteractiveApprover` rewrite

- FR-6. `InteractiveApprover` no longer writes `approval_required`
  or `tool_result` to the per-request `SSEWriter`. It writes them
  to the `SyncHub` instead.
- FR-7. The leader sees these events on the central SSE (via the
  SharedWorker), not on its POST response stream.
- FR-8. Either tab may POST `/approve` with a `requestId` — the
  server resolves whichever pending approval matches. The
  `InteractiveApprover` instance is per-turn; its resolver map is
  scoped to the originating tab's POST. Both paths still work.

### SharedWorker

- FR-9. One SharedWorker per origin. Tabs connect via
  `MessageChannel`; the worker assigns each tab a UUID on connect.
- FR-10. The worker owns the central SSE connection (one per
  origin). It reconnects with exponential backoff on disconnect
  (cap 30s, max 5 retries, then surface error to tabs).
- FR-11. On every (re)connect, the worker emits a `resync` message
  to all connected tabs. Tabs respond by reloading the active
  session transcript and the sessions list.
- FR-12. The worker has no authority over actions — tabs POST
  directly to the server with `X-CW-Tab: <uuid>`.

### Tab UUID (`X-CW-Tab`)

- FR-13. The tab UUID is generated by the SharedWorker on connect
  and stable for the lifetime of the tab's connection.
- FR-14. The tab sends `X-CW-Tab: <uuid>` on every `POST /messages`
  request.
- FR-15. The server reads `X-CW-Tab` and stamps `originator` on
  `message_appended` events with the same UUID.

### Tab reducer (`message_appended`)

- FR-16. The reducer handles `message_appended` idempotently:
  - Skip if a message with the same `id` already exists in
    `messagesBySession[sessionId]` (covers tab re-connect and any
    other duplicate).
  - Skip if `originator === state.tabId` (the originating tab
    already has the message via optimistic append).
  - Otherwise append `ev.message` to `messagesBySession[ev.sessionId]`.
- FR-17. The reducer picks up `tabId` from `state.tabId` (new field
  in `SessionsState`), set by the listener when the worker registers
  the tab.

### Resync on worker reconnect

- FR-18. On `{ kind: 'resync' }` from the worker, tabs:
  - Debounce: coalesce multiple `resync` signals within 1s.
  - Call `loadTranscript(activeSessionId)` for any active session.
  - Call `loadSessions()` to refresh the sidebar.
  - Failures are silent — next SSE event brings things in.

### Conflict semantics (existing, preserved)

- FR-19. Two tabs POSTing `/messages` simultaneously: server
  returns 409 to the loser. The loser's optimistic message is
  removed and an error banner is shown. (Existing Phase 14
  behavior.)

## Non-functional requirements

- The central SSE carries small payloads (most events are
  kilobytes), so per-event overhead is low.
- One central SSE per origin vs N tabs: strictly fewer
  connections, lower server CPU. At this project's scale (single
  user, local-first), this is negligible — but the design is
  principled.
- `bun run typecheck && bun test` is green.
- The leader's per-message SSE behavior is unchanged (Phase 14
  invariants hold).

## Out of scope

- **Multi-machine cross-tab sync.** The central SSE architecture
  extends naturally; V1 only ships same-machine/multi-tab.
- **Multi-user / shared workspaces.** Single-user remains the
  model.
- **Live token streaming across tabs.** The leader's POST keeps
  live tokens leader-only; passive viewers see messages after the
  turn completes (via `message_appended`). Acceptable for V1.
- **Service Worker / offline support.** SharedWorker is
  browser-only; the SSE is the durable channel.
- **Server-side cursor-based event replay (the V2 path).** V1
  relies on tab-side `loadTranscript` + `loadSessions` on resync;
  if "no flicker ever" becomes a requirement, the next step is
  server-side sequence numbers + `Last-Event-Id`-style replay on
  reconnect. Out of scope for V1.

## Constraints

- Same single-user / single-machine model as the rest of
  ComputerWorks.
- The SharedWorker + central SSE pattern requires a browser that
  supports SharedWorker (most modern browsers). Fallback for very
  old browsers: each tab connects directly to `GET /api/sync`
  (one SSE per tab). Detected at boot via `typeof SharedWorker ===
  'undefined'`.

## Acceptance criteria

- `bun run typecheck && bun test` is green across the project.
- A `SyncHub` unit test covers subscribe / unsubscribe / broadcast /
  dead-writer pruning.
- A `routes/sync.test.ts` covers the GET endpoint: connection
  opens, registers, receives a broadcast, unsubscribes on close.
- The existing per-message SSE event vocabulary is unchanged
  (`message_start` / `token` / `tool_call` / `done`).
- A reducer test covers `message_appended` for the three cases
  (idempotent by id, originator self-dedupe, normal append).
- Manual smoke (two tabs at `?session=<id>`): Tab A sends,
  Tab B sees user message + assistant reply; Tab A's approval
  appears in Tab B; either tab can click Approve and both see
  `tool_result`; sidebar updates propagate; a brief drop+reconnect
  triggers `resync` and both tabs recover.
